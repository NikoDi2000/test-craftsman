# 使用 test-craftsman

## 体系概览

test-craftsman 是一套质量工程师体系，由 **3 个 Skill** 和 **4 个 Agent** 组成。Skill 定义"做什么"，Agent 定义"谁来做"。

```
┌─────────────────────────────────────────────────┐
│                   主 Agent                       │
│            （协调者，决定用哪个 Skill）             │
└──────────┬──────────┬──────────┬────────────────┘
           │          │          │
     ┌─────▼────┐ ┌──▼─────┐ ┌──▼──────────────┐
     │ ATDD     │ │  PBT   │ │ API 集成测试     │
     │ 三Agent对抗│ │ 单Agent │ │ 三Agent流水线    │
     └─────┬────┘ └──┬─────┘ └──┬──────────────┘
           │         │          │
    ┌──────┼─────┐   │   ┌──────┼──────────┐
    │      │     │   │   │      │          │
  设计师 实现者 审计员 设计师 设计师 工程师  审计员
```

## Skill 选择

| 用户意图 | 触发信号 | 推荐 Skill |
|---------|---------|-----------|
| 写测试、实现功能、修 bug、重构 | "TDD"、"测试"、"单测"、"覆盖率" | `adversarial-tdd` |
| 序列化、数据验证、算法、不变性 | "属性测试"、"PBT"、"Hypothesis" | `property-based-testing` |
| API 端点测试、HTTP 测试、认证测试 | "集成测试"、"TestClient"、"端点测试" | `api-integration-testing` |
| 不确定 | 先问用户要哪种测试 | — |

### 组合使用

Skill 可以组合。常见组合：

| 场景 | 组合 | 原因 |
|------|------|------|
| FastAPI 后端新功能 | ATDD + API 集成测试 | ATDD 覆盖单元层，API 集成测试覆盖链路层 |
| 数据验证逻辑 | ATDD + PBT | ATDD 覆盖已知场景，PBT 覆盖未知边界 |
| 完整后端质量保障 | ATDD + PBT + API 集成测试 | 三层互补 |

## Agent 团队

| Agent | 身份 | temperature | 文件权限 | 核心能力 |
|-------|------|-------------|---------|---------|
| `@测试设计师` | 破坏者 | 0.2 | 只能编辑测试文件 | 风险分析、场景设计、变异假设 |
| `@实现者` | 建设者 | 0.3 | 禁止编辑测试文件 | 最小实现、红绿循环 |
| `@测试审计员` | 审查者 | 0.1 | 完全禁止编辑 | 变异测试、覆盖检查、缺口发现 |
| `@集成测试工程师` | 链路验证者 | 0.2 | 只能编辑测试文件 | HTTP 语义、认证、安全、DB 验证 |

### 权限隔离

权限隔离是体系的核心设计，确保 Agent 之间不会互相配合：

```
@测试设计师          @实现者              @测试审计员
┌──────────┐    ┌──────────┐    ┌──────────┐
│ 编辑测试 ✓ │    │ 编辑测试 ✗ │    │ 编辑任何 ✗ │
│ 编辑代码 ✗ │    │ 编辑代码 ✓ │    │ 只读一切 ✓ │
│ 调度子Agent ✗│    │ 调度子Agent ✗│    │ 调度子Agent ✗│
└──────────┘    └──────────┘    └──────────┘
     ↑               ↑               ↑
     │               │               │
   破坏者           建设者           审查者
  "找 bug"       "让测试通过"     "测试有效吗？"
```

## 工作流

### adversarial-tdd：三 Agent 对抗

```
用户: "用 TDD 实现用户注册"
         │
         ▼
   ┌─ @测试设计师 ─────────────────────────────────┐
   │ 1. 风险分析（可测试性/接口契约/输入空间）        │
   │ 2. 设计测试场景（正常 + 痛苦路径）              │
   │ 3. 写一条失败测试                               │
   │ 4. 运行确认失败 → TESTS_READY                   │
   └──────────────┬────────────────────────────────┘
                  │ 交接: 测试文件 + 失败证据
                  ▼
   ┌─ @实现者 ─────────────────────────────────────┐
   │ 1. 只看测试代码，不看风险分析                    │
   │ 2. 写最小代码让测试通过                         │
   │ 3. 运行确认通过 → GREEN_CONFIRMED               │
   └──────────────┬────────────────────────────────┘
                  │ 交接: 测试 + 实现代码
                  ▼
   ┌─ @测试审计员 ─────────────────────────────────┐
   │ 1. 变异测试（自动化 + 手动）                     │
   │ 2. 覆盖率检查                                   │
   │ 3. 通用审计6项检查                               │
   │ 4. 判定: APPROVE 或 REJECT                      │
   └──────────────┬────────────────────────────────┘
                  │
            REJECT → 反馈给 @测试设计师 → 补充测试 → 重新循环
            APPROVE → 下一条测试（垂直切片）
```

**关键约束**：
- 测试设计师与实现者**上下文隔离**——实现者看不到风险分析，只看到测试代码
- 垂直切片：一次只推进一条测试，红→绿→审计→下一条
- 测试直接通过 = 丢弃重设计（信条1：必须看到失败）

### property-based-testing：单 Agent 属性发现

```
用户: "用属性测试验证排序函数"
         │
         ▼
   ┌─ @测试设计师 ─────────────────────────────────┐
   │ 1. 三个问题发现属性：                           │
   │    - 有什么不变量？（排序后长度不变）             │
   │    - 有什么可逆操作？（排序再排序不变）           │
   │    - 有什么等价关系？（排序结果 = 参考实现结果）  │
   │ 2. 用 Hypothesis/glados/FsCheck 编写属性测试    │
   │ 3. 运行确认有反例或覆盖充分                      │
   └───────────────────────────────────────────────┘
```

**注意**：PBT 模式下只有测试设计师工作，不需要实现者和审计员。属性测试本身通过随机输入发现反例，天然具备"对抗性"。

### api-integration-testing：三 Agent 流水线

```
用户: "为用户端点写集成测试"
         │
         ▼
   ┌─ @测试设计师 ─────────────────────────────────┐
   │ 1. 测试设计六步法：                             │
   │    可测试性 → HTTP 语义 → 认证矩阵 → 安全清单   │
   │    → 数据库验证                                 │
   │ 2. 输出测试场景清单                             │
   └──────────────┬────────────────────────────────┘
                  │ 交接: 场景清单
                  ▼
   ┌─ @集成测试工程师 ─────────────────────────────┐
   │ 1. 按场景编写集成测试代码                       │
   │ 2. 验证 HTTP 状态码、响应体、数据库状态          │
   │ 3. 安全测试（BOLA、注入、数据暴露）              │
   │ 4. 运行确认通过 → INTEGRATION_TESTS_READY       │
   └──────────────┬────────────────────────────────┘
                  │ 交接: 测试代码 + 覆盖报告
                  ▼
   ┌─ @测试审计员 ─────────────────────────────────┐
   │ 1. HTTP 语义覆盖审查                            │
   │ 2. 认证场景覆盖审查                             │
   │ 3. 安全测试覆盖审查                             │
   │ 4. 测试隔离性审查                               │
   │ 5. 判定: APPROVE 或 REJECT                      │
   └──────────────┬────────────────────────────────┘
                  │
            REJECT → 反馈给 @测试设计师 → 补充场景
            APPROVE → 完成
```

**与 ATDD 的区别**：
- ATDD 是**对抗**模型：设计师 vs 实现者，互相博弈
- API 集成测试是**流水线**模型：设计师 → 工程师 → 审计员，顺序推进
- API 集成测试没有"实现者"——因为测试的是已有 API，不需要写实现

## 交接信号

Agent 之间通过标准化信号交接：

| 信号 | 发送者 | 含义 | 接收者应做什么 |
|------|--------|------|--------------|
| `TESTS_READY` | 测试设计师 | 测试已写好并确认失败 | 实现者开始实现 |
| `RED_CONFIRMED` | 实现者 | 实现前测试确实失败 | —（记录在案） |
| `GREEN_CONFIRMED` | 实现者 | 实现后测试全部通过 | 审计员开始审计 |
| `TESTS_ALREADY_PASSING` | 实现者 | 测试无需实现就通过 | 主 Agent 决定：丢弃测试或重设计 |
| `INTEGRATION_TESTS_READY` | 集成测试工程师 | 集成测试已通过 | 审计员开始审计 |
| `SECURITY_FINDINGS` | 集成测试工程师 | 发现安全问题 | 主 Agent 评估严重性 |
| `APPROVE` | 测试审计员 | 测试有效，无缺口 | 推进下一条测试或完成 |
| `REJECT` | 测试审计员 | 发现缺口 | 反馈给测试设计师补充 |
| `ESCALATE_TO_AUDIT` | 测试设计师/工程师 | 需要审计 | 审计员介入 |

## Agent 模式切换

同一个 Agent 在不同 Skill 下切换工作模式：

### @测试设计师 的三种模式

| 模式 | 触发 Skill | 分析框架 | 输出 |
|------|-----------|---------|------|
| ATDD 模式 | `adversarial-tdd` | 风险矩阵（可测试性/接口契约/输入空间/场景/变异假设/预言） | 失败测试 + 风险文档 |
| API 集成模式 | `api-integration-testing` | 六步法（可测试性/HTTP 语义/认证矩阵/安全清单/DB 验证） | 场景清单 |
| PBT 模式 | `property-based-testing` | 三问题（不变量/可逆操作/等价关系） | 属性测试 |

### @测试审计员 的两种模式

| 模式 | 触发 Skill | 审计重点 | 参考文档 |
|------|-----------|---------|---------|
| ATDD 审计 | `adversarial-tdd` | 变异测试、覆盖率、通用6项 | `adversarial-tdd/references/03-变异测试清单.md` |
| API 集成审计 | `api-integration-testing` | HTTP 语义、认证、安全、隔离 | `api-integration-testing/SKILL.md` 六步法 |

## 跨 Skill 协作

### ATDD + API 集成测试（FastAPI 后端推荐）

```
用户: "实现用户注册 API"
         │
    ┌────┴────┐
    │  ATDD   │  单元层：测试设计师写单元测试 → 实现者实现 → 审计员审计
    │  先行   │  覆盖：业务逻辑、数据验证、错误处理
    └────┬────┘
         │ 单元测试通过后
    ┌────┴──────────┐
    │ API 集成测试   │  集成层：设计师分析端点 → 工程师写集成测试 → 审计员审计
    │    后行        │  覆盖：HTTP 链路、认证、安全、DB 状态
    └───────────────┘
```

**为什么 ATDD 先行**：单元测试快速验证核心逻辑正确，集成测试验证组装后的链路正确。先跑快的，再跑慢的。

### ATDD + PBT（数据验证逻辑推荐）

```
用户: "实现数据验证逻辑"
         │
    ┌────┴────┐
    │  ATDD   │  已知场景：边界值、空值、格式错误
    │  覆盖   │  人工能想到的所有情况
    └────┬────┘
         │
    ┌────┴────┐
    │   PBT   │  未知场景：框架随机生成输入
    │  补充   │  发现人工遗漏的边界和反例
    └─────────┘
```

**为什么互补**：ATDD 覆盖"我知道会出错的地方"，PBT 覆盖"我不知道会出错的地方"。

## 常见问题

| 问题 | 回答 |
|------|------|
| 可以跳过审计员吗？ | 不可以。审计员是质量门禁，跳过等于放弃质量保证 |
| 实现者可以看测试设计师的风险分析吗？ | 不可以。上下文隔离是核心设计，防止实现者"配合"测试 |
| PBT 为什么不需要审计员？ | 属性测试的"对抗性"来自随机输入，天然发现反例 |
| API 集成测试为什么没有实现者？ | 集成测试验证已有 API，不需要写实现代码 |
| 可以同时触发多个 Skill 吗？ | 可以，但建议按 ATDD → PBT → API 集成测试 的顺序 |
| 测试设计师在 ATDD 和 API 集成测试中有什么不同？ | ATDD 模式用风险矩阵，API 模式用六步法；分析框架不同，输出不同 |
