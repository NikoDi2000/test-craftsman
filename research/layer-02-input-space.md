# Layer 2 深度报告：输入空间建模

---

## 一、理论基础：为什么输入空间建模是测试的核心

### 1.1 问题的本质

一个函数 `f(x)` 接受输入 `x`，产生输出 `f(x)`。测试的本质是选择一组输入 `{x₁, x₂, ..., xₙ}`，使得执行这些输入后，我们对 `f` 的正确性有足够的信心。

**核心挑战**：对于任何非平凡函数，输入空间是无穷大或天文数字级别的。如何从中选择有限数量的测试输入？

所有输入空间建模方法都在回答同一个问题：**哪些输入是"相似的"（可以互相代表），哪些是"不同的"（必须分别测试）？**

### 1.2 输入空间建模的历史脉络

```
1979: Myers 提出等价类划分 + 边界值分析
1988: Ostrand & Balcer 提出 Category-Partition 方法 + TSL
1993: Grochtmann & Grimm 提出 Classification Tree Method (CTM)
1999: NIST 开始研究交互故障分布
2006: NIST 发布 ACTS 工具
2024: LLM + 输入空间建模的融合研究
```

## 二、等价类划分

### 2.1 Myers 的原始定义（1979）

等价类划分的核心思想：**如果程序对一个输入的响应"代表"了它对一个类中所有其他输入的响应，那么这些输入就属于同一个等价类。**

Myers 定义了两种等价类：

| 类型 | 定义 | 例子 |
|------|------|------|
| **有效等价类** | 合法、有意义的输入 | 年龄 0-150 之间的整数 |
| **无效等价类** | 非法、无意义的输入 | 年龄 -5、年龄 "abc"、年龄 999 |

**指导原则**：
1. 如果输入条件指定了**范围**，划分一个有效类 + 两个无效类（低于范围、高于范围）
2. 如果输入条件指定了**集合**，每个有效值一个类 + 一个无效类（不在集合中的值）
3. 如果输入条件是**必须**（如"第一个字符必须是字母"），一个有效类 + 一个无效类

### 2.2 Cem Kaner 的领域测试理论

Kaner 在 BBST（Black Box Software Testing）课程中对等价类进行了更深层次的定义：

> "Two tests are equivalent relative to a potential error if both would trigger that error, or neither would trigger that error."

这个定义的革命性在于：

1. **等价性是相对于错误的**——不是相对于输入本身的相似性
2. **不同的错误模式产生不同的等价类划分**——没有"绝对"的等价类
3. **你需要先假设可能出什么错，再划分等价类**

```python
def calculate_discount(age: int, is_member: bool) -> float:
    """计算折扣。老年人(>=65) 30%，会员 10%，两者可叠加。"""
    discount = 0.0
    if age >= 65:
        discount += 0.30
    if is_member:
        discount += 0.10
    return min(discount, 1.0)
```

**按"范围错误"划分等价类**：
- 年龄 < 0（无效）
- 年龄 0-64（无老年折扣）
- 年龄 >= 65（有老年折扣）
- 年龄 > 150（可能无效）

**按"叠加错误"划分等价类**：
- 非老年人 + 非会员 → 0%
- 非老年人 + 会员 → 10%
- 老年人 + 非会员 → 30%
- 老年人 + 会员 → 40%（叠加）

两种划分方式产生不同的测试用例——因为它们针对不同的潜在错误。

### 2.3 等价类划分的局限性

等价类划分的问题在于：**它是一种启发式方法，没有形式化的保证。**

- 没有证明"等价"的数学方法
- 人为划分的主观性
- 单维度划分忽略了参数交互

## 三、Category-Partition 方法（Ostrand & Balcer, 1988）

### 3.1 为什么 CPM 是里程碑

CPM 是第一个**形式化、工具支持、系统化**的功能测试生成方法。被引用 900+ 次，是后续所有形式化测试规格方法的鼻祖。

### 3.2 CPM 的八步法

```
步骤 1: 将功能规格分解为可独立测试的功能单元
步骤 2: 识别参数和环境条件
步骤 3: 为每个参数/条件找到 Categories（分类维度）
步骤 4: 将每个 Category 划分为 Choices（具体选择）
步骤 5: 添加约束条件（消除不可能的 Choice 组合）
步骤 6: 编写 TSL（Test Specification Language）规格
步骤 7: 用工具生成 Test Frames
步骤 8: 将 Test Frames 实例化为具体测试用例
```

### 3.3 一个完整的 CPM 示例

假设测试一个搜索功能，输入参数为：
- `pattern`：搜索模式
- `case_sensitive`：是否区分大小写
- `max_results`：最大结果数

```
Categories:
    Pattern size:
        empty              [property Empty]
        single character   [property NonEmpty]
        many characters    [property NonEmpty]
        very long (>1000)  [property NonEmpty]
        special chars      [property NonEmpty]
        unicode            [property NonEmpty]

    Pattern matching:
        exact match exists
        partial match exists
        no match exists

    Case sensitivity:
        case sensitive
        case insensitive     [property CaseInsensitive]

    Max results:
        zero               [error]
        one
        many
        negative            [error]
        very large          [error]

    Pattern relation to case:
        all lower case      [if !CaseInsensitive]
        all upper case      [if !CaseInsensitive]
        mixed case          [if !CaseInsensitive]
```

**关键创新**：
- `[property X]` 标记允许选择之间传递信息
- `[if X]` 选择器允许约束条件
- `[error]` 标记让工具知道这是预期产生错误的测试
- `[single]` 标记表示这个选择不参与组合

### 3.4 TSL 的形式化语法

TSL 是一种形式化语言，用于描述 Categories、Choices 和约束：

```
# TSL 语法（简化版）

Categories:
    Category_Name:
        Choice_1. [property Prop1, Prop2]
        Choice_2. [if Prop1] [error]
        Choice_3. [single]
```

TSL Compiler 读取 TSL 文件，自动生成 Test Frames。每个 Test Frame 是从每个 Category 中选择一个 Choice 的组合，且满足所有约束条件。

### 3.5 Choice Relation Framework（Chen, Poon & Tse, 2003）

对 CPM 的重要改进，发表在 IEEE TSE 上：

- 原始 CPM 的选择关系是**二元的**（选择要么可以组合，要么不可以）
- CRF 引入了**部分关系**——某些选择在特定条件下可以共存
- 这大幅减少了无效 Test Frames 的数量

## 四、Classification Tree Method（Grochtmann & Grimm, 1993）

### 4.1 CTM 的核心思想

CTM 与 CPM 类似，但使用**树状可视化表示**，最初由 Daimler-Benz 为嵌入式系统测试开发。

```
分类树表示法：

                    测试对象
                   /   |   \
              速度   档位   模式
             / | \   /|\   /  \
           低 中 高  P R N D  运动 经济 舒适
```

### 4.2 CTM 的工作流程

1. **识别测试相关方面**（Classification）：基于功能规格，确定对测试重要的维度
2. **划分类**（Classes）：每个 Classification 划分为互斥的、完整的 Classes
3. **构建分类树**：用树状图可视化所有 Classification 和 Classes
4. **构建组合表**：选择 Class 组合形成测试用例

### 4.3 CTM 相比 CPM 的优势

| 维度 | CPM | CTM |
|------|-----|-----|
| 表示方式 | 文本（TSL）| 图形化树 |
| 可读性 | 需要理解 TSL 语法 | 直观可视化 |
| 工业应用 | 较少 | 广泛应用于航空、汽车、铁路电子 |
| 工具 | TSLgenerator | CTE/CTE XL |

### 4.4 CTE XL 的扩展（DaimlerChrysler, 2000）

CTE XL 在原始 CTM 基础上增加了：

1. **逻辑依赖规则**：Class A 需要 Class B 存在或不存在
2. **优先级标记**：区分高/中/低优先级的 Classification
3. **自动测试用例生成**：基于树的自动组合
4. **数值约束**：对数值参数的范围限制

### 4.5 CTM 的最新发展：CTM Shield（2017）

传统 CTM 的约束表达过于冗长。Kitamura et al. 提出了"参数屏蔽"方法——用参数而不是类来表达约束，将约束长度平均减少 14%。

## 五、组合测试

### 5.1 交互规则

NIST 的 Kuhn 等人研究了多个大型软件项目的故障数据，发现了一个惊人的规律——**交互规则（Interaction Rule）**：

```
1-way 交互（单个参数值）触发 ~67% 的故障
2-way 交互触发 ~93% 的故障
3-way 交互触发 ~98% 的故障
4-6 way 交互触发 ~100% 的故障
```

这个发现意味着：**穷举测试不必要。测试所有 t-way 组合（t ≤ 6）几乎等于穷举测试。**

### 5.2 验证交互规则的数据来源

NIST 在多个领域验证了这一规律：

| 应用 | 数据类型 | 最大交互度 |
|------|---------|-----------|
| 医疗器械召回 | 15 年召回数据分析 | 4-way |
| NASA 分布式数据库 | 生产环境故障 | 4-way |
| 浏览器安全漏洞 | 安全审计报告 | 5-way |
| 网络服务器 | 日志分析 | 4-way |
| MySQL | Bug 数据库 | 5-way |

### 5.3 Covering Array（覆盖数组）

覆盖数组是组合测试的数学基础：

```
CA(N; t, k, v)

其中：
  N = 测试用例数量
  t = 覆盖强度（几-way）
  k = 参数数量
  v = 每个参数的取值数

一个 CA(N; t, k, v) 保证了：对于任意 t 个参数，
它们的每一种取值组合至少出现在 N 个测试用例中的一个里。
```

**实例**：

```
CA(9; 2, 4, 3)  —  9 个测试用例覆盖 4 个参数
                   （每个参数 3 个值）的所有 2-way 组合

参数: A={a1,a2,a3}, B={b1,b2,b3}, C={c1,c2,c3}, D={d1,d2,d3}

测试 1: a1, b1, c1, d1
测试 2: a1, b2, c2, d2
测试 3: a1, b3, c3, d3
测试 4: a2, b1, c2, d3
测试 5: a2, b2, c3, d1
测试 6: a2, b3, c1, d2
测试 7: a3, b1, c3, d2
测试 8: a3, b2, c1, d3
测试 9: a3, b3, c2, d1

验证：A×B 的所有 9 种组合都至少出现一次 ✓
      A×C 的所有 9 种组合都至少出现一次 ✓
      ... (所有 6 个 2-参数对)
```

如果穷举测试所有 4 个参数的所有值组合，需要 3⁴ = 81 个测试。而 9 个测试就能覆盖所有 2-way 交互。

### 5.4 主要工具

| 工具 | 开发者 | 特点 |
|------|--------|------|
| **ACTS** | NIST | 工业级，支持约束，混合强度覆盖 |
| **PICT** | Microsoft | 命令行，支持复杂约束语法 |
| **CITlab** | 多个研究机构 | 可插拔的生成器框架 |
| **Jenny** | Bob Jenkins | 轻量级，适合简单场景 |

### 5.5 约束处理

现实世界的参数有约束——某些值组合不可能或不应该出现。

**PICT 的约束语法**：

```
# 参数定义
OS: Windows, macOS, Linux
Browser: Chrome, Firefox, Safari, Edge

# 约束
IF [OS] = "macOS" THEN [Browser] <> "Edge";
IF [OS] = "Linux" THEN [Browser] IN {"Chrome", "Firefox"};
```

**ACTS 的混合强度覆盖**：

不是所有参数交互都同等重要。ACTS 支持为不同的参数子集指定不同的覆盖强度：

```
所有参数对: t=2（pairwise）
{Encryption, Protocol}: t=3（这 3 个参数的 3-way）
{Authentication, Session}: t=4（这 2 个参数的 4-way）
```

### 5.6 组合测试 vs 随机测试

NIST & Microsoft Research 2014 年的对比实验（Siemens Suite）：

**发现**：
- 在大多数情况下，t-way 测试表现等于或略优于随机测试
- 差异不大——说明**随机测试的自然 t-way 覆盖率已经很高**
- 但 t-way 测试的**稳定性**更好（多次运行的效果更一致）

**实践意义**：如果你能系统建模参数和约束，组合测试更好。如果时间紧迫，有足够样本量的随机测试也不错。

## 六、边界值分析（Boundary Value Analysis）

### 6.1 经典边界值

Myers 的经典边界值测试：

```
对于范围 [min, max]：
  min-1, min, min+1, max-1, max, max+1

对于值集合：
  集合最小值, 集合最大值, 略小于最小值, 略大于最大值
```

### 6.2 多维边界值

Arnicane (2008) 系统化了 17 种边界值测试方法。核心发现：

**单维边界值测试不足以发现多维交互的边界错误。**

多维边界值：
```
(min, min), (min, max), (max, min), (max, max)
加上 "刚好越过边界" 的组合：
(min-1, max), (min, max+1), ...
```

### 6.3 边界值的类型

边界不一定是数值的：

| 边界类型 | 例子 |
|---------|------|
| **数值边界** | 0, 最大值, 最小值, 负数 |
| **大小边界** | 空字符串, 空列表, 单元素, 满容量 |
| **时间边界** | 00:00, 23:59, 闰年, 时区切换 |
| **结构边界** | 空树, 单节点树, 退化的树 |
| **并发边界** | 0 个并发请求, 1 个, 最大并发数 |
| **权限边界** | 未认证, 最低权限, 最高权限 |
| **状态边界** | 初始状态, 中间状态, 最终状态 |

### 6.4 边界值分析对 AI 的启示

AI 在做边界值分析时最容易犯的错误是**只识别数值边界**。一个完整的边界值分析应该覆盖上述所有类型。

## 七、域测试

### 7.1 Cem Kaner 的域测试理论

域测试将输入空间建模为一个多维空间，每个维度是一个输入参数。

**基本概念**：

```
输入空间 = 所有可能的输入组合
域 = 输入空间的一个子集（如"合法输入"）
边界 = 域之间的界面（错误最容易发生的地方）
```

**域测试的核心思想**：
- 测试**域的边界**比测试**域的内部**更有价值
- 边界是错误的高发区——因为程序员容易混淆边界条件

### 7.2 域的几何模型

对于二维输入空间 `(x, y)`，域可能是：

```
线性边界：y > 2x + 1
矩形边界：10 < x < 20, 5 < y < 15
非线性边界：x² + y² < 100
```

**测试策略**：
- **ON 点**：恰好落在边界上的点
- **OFF 点**：刚好在边界外的点
- **IN 点**：深入域内部的点

### 7.3 多维域测试的挑战

随着维度增加：
- 边界数量呈指数增长
- OFF 点的选择变得模糊（对于"刚好在边界外"的几何位置）
- 需要组合测试来覆盖多维边界

## 八、AI 驱动的输入空间建模

### 8.1 LLM 可以做什么

| 任务 | LLM 的能力 | 局限 |
|------|-----------|------|
| **参数识别** | 从函数签名 + 文档识别参数 | 对隐式参数（全局状态）可能遗漏 |
| **等价类划分** | 基于语义推理划分有效/无效类 | 缺少对"相对什么错误"的推理 |
| **边界值推导** | 从参数名和类型推导边界 | 对业务逻辑边界（如"VIP 会员阈值"）依赖上下文 |
| **约束发现** | 从自然语言规格中提取约束 | 隐式约束（如"如果 x 存在，y 就不能是 z"）难以提取 |

### 8.2 一种可行的 AI 输入空间建模流程

```
1. 参数提取
   输入：函数签名 + 文档字符串 + 业务需求描述
   输出：参数列表 + 每个参数的类型和语义

2. 等价类划分（每个参数独立）
   输入：参数信息
   输出：{参数 → [有效类列表, 无效类列表]}

3. 边界值推导（每个参数独立）
   输入：参数信息和等价类
   输出：{参数 → [边界值列表]}

4. 参数关系推断
   输入：所有参数信息 + 业务需求
   输出：约束列表（IF-THEN 规则）

5. 测试组合生成
   输入：等价类 + 边界值 + 约束
   输出：测试用例组合（pairwise 或 n-wise）

6. 人类审查
   输入：生成的测试组合
   输出：修正和补充
```

### 8.3 对你的 PydanticAI / FastAPI / Flutter / Unity 的应用

| 项目 | 输入空间建模重点 |
|------|---------------|
| **FastAPI** | API 参数（query, path, body）的等价类 + 边界值 + 无效值 |
| **PydanticAI** | prompt 输入空间（长度、语言、话题、恶意内容）+ 输出结构约束 |
| **Flutter** | Widget 状态空间（加载中/空数据/错误/正常）× 用户交互 |
| **Unity** | 游戏状态空间（生命值、位置、速度、碰撞）× 输入事件 |

## 九、总结：输入空间建模的核心原则

| # | 原则 | 说明 |
|---|------|------|
| 1 | **等价类划分要先问"相对什么错误"** | 不同的错误假设产生不同的划分 |
| 2 | **边界值分析要覆盖所有维度** | 数值、大小、时间、结构、并发、权限、状态 |
| 3 | **组合测试用 t-way 而非穷举** | t=2 覆盖 93%，t=4 覆盖 ~100% |
| 4 | **约束条件必须显式表达** | 不可能/不应该出现的参数组合 |
| 5 | **输入空间建模是迭代过程** | 发现新 bug → 更新模型 → 生成新测试 |
| 6 | **对 AI 来说，模型比直觉更可靠** | 显式建模输入空间 > 依赖 AI "想到"边缘情况 |
