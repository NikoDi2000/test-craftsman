# Layer 7 深度报告：认知循环

---

## 一、测试的认知本质

### 1.1 测试不是执行步骤，是问题解决

Enoiu et al. (2020-2023) 通过口头协议分析（让测试人员边测试边说）发现：

> **测试 ≠ 执行测试用例。测试 = 解决一个开放式问题："给定这个系统，什么输入能暴露它的缺陷？"**

这一洞察颠覆了传统对测试的理解：

| 传统理解 | 认知理解 |
|---------|---------|
| 测试是"按步骤执行" | 测试是"持续推理和决策" |
| 测试人员是"执行者" | 测试人员是"问题解决者" |
| 测试用例是"指令" | 测试用例是"假设" |
| 测试结果是"通过/失败" | 测试结果是"新的信息" |

### 1.2 四阶段认知循环模型

Enoiu 等人对 38 位瑞典测试工程师的调查建立了四阶段模型：

```
  ┌─────────────────────────────────────────────────────────┐
  │                    认知循环                              │
  │                                                         │
  │  理解目标 ──────→ 规划策略 ──────→ 执行测试 ──────→ 检查结果 │
  │      ↑                                                │  │
  │      └──────────────── (学习) ────────────────────────┘  │
  │                                                         │
  └─────────────────────────────────────────────────────────┘
```

| 阶段 | 测试人员在做什么 | 认知活动 |
|------|----------------|---------|
| **理解目标** | 阅读规格、问开发人员、查看代码变更、理解业务背景 | 信息收集、模式识别、假设生成 |
| **规划策略** | 等价类划分、边界值分析、基于风险的选择、优先级排序 | 分析、推理、决策 |
| **执行测试** | 运行测试、记录结果、观察意外行为、记录想法 | 观察、注意、记录 |
| **检查结果** | 判断是 bug 还是预期行为、决定下一步 | 评估、判断、反思 |

## 二、探索性测试

### 2.1 James Bach 的定义

James Bach（探索性测试的奠基人）给出了最精确的定义：

> **"Simultaneous learning, test design, and test execution."**

> "同时学习、设计测试和执行测试。"

**关键区别**：

```
脚本化测试 = 先设计全部测试，后执行（像按菜谱做菜）
探索性测试 = 边做边学边调整（像侦探调查案件）
```

### 2.2 探索性测试的 OODA 循环

探索性测试的核心是 OODA 循环（Observe-Orient-Decide-Act）：

```
观察 (Observe)
  ↓  看到了什么？什么行为是意外的？
推断 (Orient)
  ↓  这意味着什么？可能的原因是什么？
决策 (Decide)
  ↓  接下来测什么？哪个方向最可能发现 bug？
行动 (Act)
  ↓  执行测试
  └→ 观察（循环继续）
```

**OODA 循环的加速是测试专家和新手的核心区别。**

专家：
- 观察更精准（能注意到"不对劲"的细节）
- 推断更快（丰富的模式匹配经验）
- 决策更果断（知道什么方向值得探索）
- 循环更快（在相同时间内完成更多轮探索）

### 2.3 Cem Kaner 的探索性测试维度

Kaner 定义了探索性测试的五个维度：

| 维度 | 含义 | 如何培养 |
|------|------|---------|
| **测试覆盖** | 测试覆盖了系统的哪些方面 | 使用覆盖大纲（如 SFDPOT） |
| **预言** | 用什么标准判断正确性 | 多层预言策略 |
| **过程** | 测试活动的结构和记录 | 会话式测试管理（SBTM） |
| **风险** | 关注什么可能出错 | 风险分析矩阵 |
| **技术** | 使用的测试技术和工具 | 持续学习新工具 |

## 三、人类测试专家的认知优势

### 3.1 五个认知支柱

学术研究识别出人类测试专家的五个认知支柱：

| 支柱 | 人类怎么做 | AI 的差距 |
|------|----------|---------|
| **批判性思维** | 不信任需求文档，持续追问"为什么"、"如果...呢" | AI 默认信任 prompt，不质疑 |
| **怀疑精神** | 以证据为基础的怀疑，不盲目相信代码正确 | AI 没有怀疑机制 |
| **风险聚焦** | 基于"什么可能出错"来分配精力，高风险区域花更多时间 | AI 均匀分配精力 |
| **共情能力** | 想象真实用户会怎么操作、怎么犯错 | AI 没有用户视角 |
| **缺陷直觉** | 基于历史经验的模式匹配——"这种代码以前出过类似 bug" | AI 没有经验积累 |

### 3.2 专家和新手的区别

| 维度 | 新手 | 专家 |
|------|------|------|
| **等价类划分** | 只划分正常/异常 | 基于错误模式的多维划分 |
| **边界值** | 只看数值边界 | 时间、结构、状态、权限边界 |
| **探索策略** | 随机尝试 | 基于假设的定向探索 |
| **异常检测** | "看起来不对" | 精确描述"哪里不对、为什么不对" |
| **学习速度** | 缓慢积累 | 模式匹配加速学习 |

## 四、缺陷直觉（Bug Intuition）

### 4.1 缺陷直觉的本质

缺陷直觉不是神秘的天赋，而是**基于经验的无意识模式匹配**。

```
看到这种代码 → 以前出过类似 bug → 这里很可能也有问题

if (user.is_admin || user.has_permission("delete")) {
    // 以前在类似的权限检查中发现了 bypass 漏洞
    // 直觉：这里可能也有问题
    delete_record(id);
}
```

### 4.2 缺陷直觉的积累方式

| 来源 | 说明 |
|------|------|
| **历史 bug** | 自己或团队以前发现的 bug 模式 |
| **代码审查** | 在审查中见过的常见错误 |
| **安全研究** | CVE、OWASP Top 10 等已知漏洞模式 |
| **语言陷阱** | 语言特定的常见陷阱（Python 可变默认参数、JS 类型强制等） |
| **领域知识** | 业务领域的常见错误（金融：精度丢失、时间：时区） |

### 4.3 缺陷直觉对 AI 的启示

AI 可以通过以下方式积累"缺陷直觉"：

1. **代码模式库**：常见 bug 模式 → 匹配当前代码
2. **历史 bug 库**：以前发现的 bug → 类似代码告警
3. **安全知识库**：CWE、OWASP → 自动匹配

## 五、从反馈中学习：强化学习 + 测试

### 5.1 Apple 的 RL + Agentic RAG (2024)

Apple 的研究提出了用强化学习让 AI Agent 从测试反馈中学习：

```
执行 AI 生成的测试 → 获取反馈 → 更新 Agent 策略 → 更好的测试
                     (发现 bug?)
                     (误报?)
                     (覆盖率?)
```

**效果**：
- 测试生成准确率：94.8% → 97.2%（+2.4%）
- 缺陷检测率提高 10.8%

### 5.2 反馈循环的类型

| 反馈类型 | 说明 | 学习信号 |
|---------|------|---------|
| **覆盖反馈** | 测试覆盖了哪些代码 | 未覆盖代码 → 需要新测试 |
| **变异反馈** | 测试杀死了哪些变异体 | 存活变异 → 测试盲点 |
| **Bug 反馈** | 测试发现了哪些 bug | 发现 bug → 这类测试有效 |
| **误报反馈** | 测试误报了多少次 | 误报 → 测试设计有问题 |
| **维护反馈** | 测试在重构时碎了多少次 | 碎了 → 测试太脆弱 |

### 5.3 持续改进循环

```
┌──────────────────────────────────────────────────────┐
│                AI 测试的持续改进循环                    │
│                                                      │
│  1. 生成测试                                          │
│       ↓                                              │
│  2. 执行测试 + 收集反馈                                │
│       ├── 覆盖率数据                                   │
│       ├── 变异得分                                     │
│       ├── 发现的 bug                                  │
│       └── 误报统计                                     │
│       ↓                                              │
│  3. 分析反馈                                          │
│       ├── 哪些测试有效？为什么？                         │
│       ├── 哪些测试无效？为什么？                         │
│       └── 什么模式导致了无效测试？                       │
│       ↓                                              │
│  4. 改进策略                                          │
│       ├── 更新提示词模板                                │
│       ├── 调整预言选择策略                              │
│       └── 丰富边界值知识库                              │
│       ↓                                              │
│  5. 回到步骤 1（用改进后的策略）                         │
│                                                      │
└──────────────────────────────────────────────────────┘
```

## 六、对 adversarial-tdd 的启示

### 6.1 当前 adversarial-tdd 的认知缺陷

你的 adversarial-tdd 有三个 Agent，但**没有一个 Agent 有记忆**：

- 每次执行 TDD 流程，三个 Agent 都是从零开始
- 没有从历史项目中学习
- 不知道哪些测试模式有效，哪些无效
- 不会根据反馈改进策略

### 6.2 认知增强的三层架构

```
Layer 1: 即时认知（当前 adversarial-tdd 做的）
  → 基于当前代码和规格的推理

Layer 2: 会话记忆（同一个开发会话内）
  → 前一个功能的测试 → 下一个功能的测试参考
  → 已知的代码模式 → 针对性的测试策略

Layer 3: 跨项目学习（知识库）
  → 所有历史项目的 bug 模式
  → 有效的测试策略
  → 技术栈特定的陷阱
```

### 6.3 你的 QE 系统的认知循环

这正是你创建这个代码仓库的初衷——让 QE 不断学习、不断进化：

```
学习 → 内化（沉淀为 skill）→ 实践（在项目中应用）→
反思（分析哪些测试有效哪些无效）→ 优化（改进 skill）→
学习（吸收新的测试知识）
```

## 七、总结：认知循环的核心原则

| # | 原则 | 说明 |
|---|------|------|
| 1 | **测试是问题解决，不是步骤执行** | AI 需要模拟推理过程，而非生成测试用例 |
| 2 | **OODA 循环是测试的核心引擎** | 观察→推断→决策→行动→循环 |
| 3 | **专家的优势在于速度和模式匹配** | 更快的循环 + 更多的模式 = 更好的测试 |
| 4 | **没有记忆 = 没有进化** | AI 必须从每次测试执行中学习 |
| 5 | **反馈驱动改进** | 覆盖率、变异得分、bug 发现率 → 改进策略 |
| 6 | **缺陷直觉来自积累** | 代码模式库 + 历史 bug 库 + 安全知识库 |
