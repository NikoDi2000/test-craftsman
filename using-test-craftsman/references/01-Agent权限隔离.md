# Agent 权限隔离

权限隔离是 test-craftsman 体系的核心设计，确保 Agent 之间不会互相配合。

## 权限矩阵

```
@测试设计师          @实现者              @测试审计员
┌──────────┐    ┌──────────┐    ┌──────────┐
│ 编辑测试 ✓ │    │ 编辑测试 ✗ │    │ 编辑任何 ✗ │
│ 编辑代码 ✗ │    │ 编辑代码 ✓ │    │ 只读一切 ✓ │
│ 调度子Agent ✗│    │ 调度子Agent ✗│    │ 调度子Agent ✗│
└──────────┘    └──────────┘    └──────────┘
     ↑               ↑               ↑
     │               │               │
   破坏者           建设者           审查者
  "找 bug"       "让测试通过"     "测试有效吗？"
```

## 为什么隔离

同一 AI 同时写测试和实现时，测试会变成实现的影子，而非行为的守卫。学术研究表明，单 Agent TDD 的回归错误率反而比不做 TDD 更高。

隔离后：
- 测试设计师**看不到实现代码**，只能从接口出发设计测试
- 实现者**看不到风险分析**，只能从测试代码出发实现
- 审计员**不能修改任何文件**，只能审查和报告

## @测试设计师

- **可读**：所有文件
- **可编辑**：仅测试文件（`test_*.py`、`*_test.py`、`conftest.py` 等）
- **可执行**：测试运行命令（pytest、flutter test 等）、ls、cat、grep
- **不可调度**：其他 subagent（`task: deny`）

## @实现者

- **可读**：所有文件
- **可编辑**：除测试文件外的所有文件
- **可执行**：测试运行命令、git diff、ls
- **不可调度**：其他 subagent（`task: deny`）

## @测试审计员

- **可读**：所有文件
- **可编辑**：完全禁止（`edit: deny`）
- **可执行**：测试运行命令、ls、cat、grep
- **不可调度**：其他 subagent（`task: deny`）

## @集成测试工程师

- **可读**：所有文件
- **可编辑**：仅测试文件
- **可执行**：pytest、uv run pytest、ls、cat、grep
- **不可调度**：其他 subagent（`task: deny`）

## temperature 设计

| Agent | temperature | 原因 |
|-------|-------------|------|
| 测试设计师 | 0.2 | 需要系统化分析，但也要有创造性思维找边界 |
| 实现者 | 0.3 | 需要一定灵活性应对各种测试约束 |
| 测试审计员 | 0.1 | 需要最严格的审查，不应"创造性"地放过问题 |
| 集成测试工程师 | 0.2 | 需要系统化验证，但安全测试需要一定发散思维 |
