# 工作流详解

## adversarial-tdd：三 Agent 对抗

```
用户: "用 TDD 实现用户注册"
         │
         ▼
   ┌─ @测试设计师 ─────────────────────────────────┐
   │ 1. 风险分析（可测试性/接口契约/输入空间）        │
   │ 2. 设计测试场景（正常 + 痛苦路径）              │
   │ 3. 写一条失败测试                               │
   │ 4. 运行确认失败 → TESTS_READY                   │
   └──────────────┬────────────────────────────────┘
                  │ 交接: 测试文件 + 失败证据
                  ▼
   ┌─ @实现者 ─────────────────────────────────────┐
   │ 1. 只看测试代码，不看风险分析                    │
   │ 2. 写最小代码让测试通过                         │
   │ 3. 运行确认通过 → GREEN_CONFIRMED               │
   └──────────────┬────────────────────────────────┘
                  │ 交接: 测试 + 实现代码
                  ▼
   ┌─ @测试审计员 ─────────────────────────────────┐
   │ 1. 变异测试（自动化 + 手动）                     │
   │ 2. 覆盖率检查                                   │
   │ 3. 通用审计6项检查                               │
   │ 4. 判定: APPROVE 或 REJECT                      │
   └──────────────┬────────────────────────────────┘
                  │
            REJECT → 反馈给 @测试设计师 → 补充测试 → 重新循环
            APPROVE → 下一条测试（垂直切片）
```

### 关键约束

- **上下文隔离**：测试设计师与实现者上下文隔离——实现者看不到风险分析，只看到测试代码
- **垂直切片**：一次只推进一条测试，红→绿→审计→下一条
- **必须看到失败**：测试直接通过 = 丢弃重设计

### 垂直切片示例

```
切片1: test_user_registration_with_valid_data
  设计师写测试 → 确认失败 → 实现者实现 → 确认通过 → 审计员审计 → APPROVE

切片2: test_user_registration_with_duplicate_email
  设计师写测试 → 确认失败 → 实现者实现 → 确认通过 → 审计员审计 → APPROVE

切片3: test_user_registration_with_invalid_email
  ...
```

每次只推进一个切片，不批量写测试。

## property-based-testing：单 Agent 属性发现

```
用户: "用属性测试验证排序函数"
         │
         ▼
   ┌─ @测试设计师 ─────────────────────────────────┐
   │ 1. 三个问题发现属性：                           │
   │    - 有什么不变量？（排序后长度不变）             │
   │    - 有什么可逆操作？（排序再排序不变）           │
   │    - 有什么等价关系？（排序结果 = 参考实现结果）  │
   │ 2. 用 Hypothesis/glados/FsCheck 编写属性测试    │
   │ 3. 运行确认有反例或覆盖充分                      │
   └───────────────────────────────────────────────┘
```

### 为什么不需要审计员

属性测试的"对抗性"来自随机输入，天然发现反例。框架本身就是审计员——如果某个变异能被随机输入发现，Hypothesis 会找到它。

### 为什么不需要实现者

PBT 通常用于验证已有实现，不需要"先写测试再实现"的 TDD 流程。测试设计师直接写属性测试并运行。

## api-integration-testing：三 Agent 流水线

```
用户: "为用户端点写集成测试"
         │
         ▼
   ┌─ @测试设计师 ─────────────────────────────────┐
   │ 1. 测试设计六步法：                             │
   │    可测试性 → HTTP 语义 → 认证矩阵 → 安全清单   │
   │    → 数据库验证                                 │
   │ 2. 输出测试场景清单                             │
   └──────────────┬────────────────────────────────┘
                  │ 交接: 场景清单
                  ▼
   ┌─ @集成测试工程师 ─────────────────────────────┐
   │ 1. 按场景编写集成测试代码                       │
   │ 2. 验证 HTTP 状态码、响应体、数据库状态          │
   │ 3. 安全测试（BOLA、注入、数据暴露）              │
   │ 4. 运行确认通过 → INTEGRATION_TESTS_READY       │
   └──────────────┬────────────────────────────────┘
                  │ 交接: 测试代码 + 覆盖报告
                  ▼
   ┌─ @测试审计员 ─────────────────────────────────┐
   │ 1. HTTP 语义覆盖审查                            │
   │ 2. 认证场景覆盖审查                             │
   │ 3. 安全测试覆盖审查                             │
   │ 4. 测试隔离性审查                               │
   │ 5. 判定: APPROVE 或 REJECT                      │
   └──────────────┬────────────────────────────────┘
                  │
            REJECT → 反馈给 @测试设计师 → 补充场景
            APPROVE → 完成
```

### 与 ATDD 的区别

| 维度 | ATDD | API 集成测试 |
|------|------|-------------|
| 模型 | 对抗（设计师 vs 实现者） | 流水线（设计师 → 工程师 → 审计员） |
| 实现者 | 有（写生产代码） | 无（测试已有 API） |
| 测试类型 | 单元测试 | 集成测试 |
| Mock 策略 | 最小 Mock | Mock 外部服务，真实 DB |
| 核心关注 | 行为正确性 | 链路正确性 + 安全性 |
