# 对抗性TDD（ATDD）

## 概述

解决 **TDD提示悖论**：同一AI同时写测试和实现时，测试会变成实现的影子，而非行为的守卫。学术研究表明，单Agent TDD的回归错误率反而比不做TDD更高。

**核心原则**：测试设计师与实现者必须是**隔离上下文的独立Agent**。测试设计师的使命是**破坏**实现，而非配合实现。

## 测试哲学（融合自 mattpocock）

测试必须验证**行为**，而非**实现细节**：

- **好测试**：集成风格，通过公开接口验证可观测行为。代码重构时测试不碎。
- **坏测试**：Mock内部协作者、测试私有方法、断言调用次数。重构即碎。

参考 `references/04-测试哲学与原则.md` 获取详细示例。

## 何时触发

- 用户要求"写测试"
- 用户要求"实现某功能"
- 用户要求"修复bug"
- 用户要求"重构"
- 用户提到"TDD"、"测试"、"单测"、"覆盖率"
- 任何涉及代码修改的任务

## 三Agent对抗模型

| Agent | 角色 | 可见范围 | 核心使命 |
|-------|------|----------|----------|
| `@测试设计师` | 测试设计师 | 需求、接口、类型签名 | **找出所有可能的bug** |
| `@实现者` | 实现者 | 测试代码、接口契约 | **让测试通过** |
| `@测试审计员` | 测试审计员 | 测试+实现+风险分析 | **验证测试真能抓住错误** |

## 工作流

### 阶段1：需求 → 风险分析（@测试设计师）

写任何测试代码前，必须：

1. **确认公开接口**：与用户确认接口变更（参考 `references/04-测试哲学与原则.md` 中的接口设计原则）
2. **列出行为清单**：列出要测试的行为（优先关键路径和复杂逻辑，非所有边界）
3. **输出风险分析**：
   - 3条正常路径
   - 5条"痛苦场景"（空值、边界、溢出、并发、依赖失败、非法状态转换）
   - 技术栈特定追加（Flutter/FastAPI/Unity）
   - "变异假设"清单："如果实现者犯了X错误，测试Y会抓住"
4. **设计深模块**：检查是否能把复杂度隐藏在简单接口后（参考 `references/04-测试哲学与原则.md`）

**技术栈适配**：根据项目类型，加载对应的 `references/06~08-技术栈适配` 文档补充场景。

### 阶段2：RED → GREEN 垂直切片（@测试设计师 + @实现者）

**禁止水平切片**（先写所有测试再写所有实现）。必须按**垂直切片**执行：

```
RED→GREEN: test1 → impl1
RED→GREEN: test2 → impl2
RED→GREEN: test3 → impl3
...
```

每个切片内：
- `@测试设计师` 写**一条**测试，运行确认失败（因正确原因）
- `@实现者` 写**最小**代码让这条测试通过
- 重复直到行为清单完成

**门禁**：
- 如有测试直接通过，丢弃并重设计
- 实现者**禁止**修改测试文件

### 阶段3：审计（@测试审计员）

- 接收：测试文件 + 实现代码 + 原始风险分析
- 执行6项通用变异检查 + 技术栈特定变异检查（参考 `references/03-变异测试清单.md`）
- **门禁**：发现缺口 → 返回阶段2；全部通过 → 批准

### 阶段4：重构（@实现者）

- 清理代码，保持测试通过
- 寻找重构候选人：消除重复、深化模块、应用SOLID（参考 `references/05-重构指南.md`）
- **禁止在红灯时重构**
- 未经重新审计，禁止修改测试

## 安装与配置

### 第一步：配置Agent

将 `references/01-Agent配置模板.md` 中的三个Agent配置复制到：

- `.opencode/agents/测试设计师.md`
- `.opencode/agents/实现者.md`
- `.opencode/agents/测试审计员.md`

### 第二步：配置全局规则

将 `assets/全局规则模板.md` 集成到：

- `.AGENTS.md`

### 第三步：配置权限

在 `opencode.json` 中确保build agent可调度三个subagent：

```json
{
  "agent": {
    "build": {
      "permission": {
        "task": {
          "*": "allow",
          "测试设计师": "allow",
          "实现者": "allow",
          "测试审计员": "allow"
        }
      }
    }
  }
}
```

## 禁止模式（红旗）

- 测试设计师在写测试前看到了实现代码
- 实现者修改了测试文件
- 测试无需实现就直接通过
- 风险分析中缺少"痛苦场景"
- 审计员发现 >0 个未测试的变异假设
- 单个Agent同时负责测试和实现
- 水平切片（批量写测试再批量写实现）
- 测试断言实现细节（调用次数、私有方法）
