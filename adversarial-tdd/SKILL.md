# 对抗性TDD（ATDD）

## 概述

解决 **TDD提示悖论**：同一AI同时写测试和实现时，测试会变成实现的影子，而非行为的守卫。学术研究表明，单Agent TDD的回归错误率反而比不做TDD更高。

**核心原则**：测试设计师与实现者必须是**隔离上下文的独立Agent**。测试设计师的使命是**破坏**实现，而非配合实现。

## 核心信条

### 信条1：必须看到失败

> **如果你没看到测试失败，你就不知道这个测试测的是不是对的东西。**

这是 TDD 的灵魂。一个直接通过的测试毫无价值——它可能测错了东西，或者根本没测任何东西。

- **做法**：写测试后必须运行确认失败
- **门禁**：测试直接通过 → 丢弃并重设计

### 信条2：测试就是规格

> **一个好的测试读起来应该像一份规格说明书。**

测试不只是验证手段，更是活文档。`test_user_can_checkout_with_valid_cart` 比 `test_checkout_works` 更有价值——它告诉未来的开发者这个功能存在。

### 信条3：行为 > 实现

> **测试验证可观测行为，不是内部实现细节。**

- **好测试**：集成风格，通过公开接口验证行为。代码重构时测试不碎。
- **坏测试**：Mock内部协作者、测试私有方法、断言调用次数。重构即碎。

参考 `references/04-测试哲学与原则.md` 获取详细示例。

### 信条4：对抗产生质量

> **测试设计师的使命是破坏实现，而非配合实现。**

同一个大脑写测试和实现会产生盲点。隔离的测试设计师必须带着"找bug"的心态来设计测试。

## 何时触发

- 用户要求"写测试"
- 用户要求"实现某功能"
- 用户要求"修复bug"
- 用户要求"重构"
- 用户提到"TDD"、"测试"、"单测"、"覆盖率"
- 任何涉及代码修改的任务

## 三Agent对抗模型

| Agent | 角色 | 可见范围 | 核心使命 | 定义文件 |
|-------|------|----------|----------|----------|
| `@测试设计师` | 破坏者 | 需求、接口、类型签名 | **找出所有可能的bug** | `agents/测试设计师.md` |
| `@实现者` | 建设者 | 测试代码、接口契约 | **让测试通过** | `agents/实现者.md` |
| `@测试审计员` | 审查者 | 测试+实现+风险分析 | **验证测试真能抓住错误** | `agents/测试审计员.md` |

Agent 定义已提取到项目根目录 `agents/` 下，所有 Skill 共享同一套 Agent。

## 工作流

### 阶段1：需求 → 风险分析（@测试设计师）

写任何测试代码前，必须按 `references/02-风险矩阵模板.md` 完成分析：

1. **可测试性评估**：代码可控吗？可观测吗？确定吗？任一"否"→ 先重构
2. **接口契约**：前置条件 + 后置条件 + 不变量
3. **输入空间建模**：等价类划分 + 边界值 + 参数组合
4. **场景设计**：正常路径(≥3) + 痛苦场景(≥5) + 技术栈特定追加
5. **变异假设**："如果实现者犯了X错误，测试Y会抓住"
6. **预言选择**：每个场景选择正确的断言类型

**技术栈适配**：根据项目类型，加载对应的 `references/06~08-技术栈适配` 文档补充场景。

### 阶段2：RED → GREEN 垂直切片（@测试设计师 + @实现者）

**禁止水平切片**（先写所有测试再写所有实现）。必须按**垂直切片**执行：

```
RED→GREEN: test1 → impl1
RED→GREEN: test2 → impl2
RED→GREEN: test3 → impl3
...
```

每个切片内：
- `@测试设计师` 写**一条**测试，运行确认失败（因正确原因）
- `@实现者` 写**最小**代码让这条测试通过
- 重复直到行为清单完成

**门禁**：
- 如有测试直接通过，丢弃并重设计
- 实现者**禁止**修改测试文件

### 阶段3：审计（@测试审计员）

- 接收：测试文件 + 实现代码 + 原始风险分析
- 路径覆盖检查：运行覆盖率工具，分析未覆盖分支
- 自动化变异测试：运行变异测试工具，变异得分 ≥ 70%
- 手动变异检查：6项通用 + 技术栈特定变异
- **门禁**：发现缺口 → 存活变异体反馈给测试设计师 → 返回阶段2；全部通过 → 批准
- 参考 `references/03-变异测试清单.md`

### 阶段4：重构（@实现者）

- 清理代码，保持测试通过
- 寻找重构候选人：消除重复、深化模块、应用SOLID（参考 `references/05-重构指南.md`）
- **禁止在红灯时重构**
- 未经重新审计，禁止修改测试

## 安装与配置

### 第一步：配置Agent

将 `agents/` 目录下的三个 Agent 配置复制到：

- `.opencode/agents/测试设计师.md`（源文件：`agents/测试设计师.md`）
- `.opencode/agents/实现者.md`（源文件：`agents/实现者.md`）
- `.opencode/agents/测试审计员.md`（源文件：`agents/测试审计员.md`）

Agent 定义在项目根目录 `agents/` 下，所有 Skill 共享同一套 Agent。

### 第二步：配置全局规则

将 `assets/全局规则模板.md` 集成到：

- `.AGENTS.md`

### 第三步：配置权限

在 `opencode.json` 中确保build agent可调度三个subagent：

```json
{
  "agent": {
    "build": {
      "permission": {
        "task": {
          "*": "allow",
          "测试设计师": "allow",
          "实现者": "allow",
          "测试审计员": "allow"
        }
      }
    }
  }
}
```

## 禁止模式（红旗）

- 测试设计师在写测试前看到了实现代码
- 实现者修改了测试文件
- 测试无需实现就直接通过
- 风险分析中缺少"痛苦场景"
- 审计员发现 >0 个未测试的变异假设
- 单个Agent同时负责测试和实现
- 水平切片（批量写测试再批量写实现）
- 测试断言实现细节（调用次数、私有方法）
