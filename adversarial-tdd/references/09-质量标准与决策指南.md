# 测试质量标准与决策指南

## 好测试的特征

| 标准 | 说明 |
|------|------|
| **描述行为** | 测试名描述"做什么"而非"怎么做" |
| **一个行为一个测试** | 每个测试只验证一个行为 |
| **不依赖顺序** | 测试可以任意顺序运行 |
| **隔离** | 不共享状态，不依赖其他测试 |
| **读起来像规格** | `test_user_can_checkout_with_valid_cart` |

## Test Smells（测试异味）

如果出现以下情况，测试质量不合格：

| Smell | 说明 | 例子 |
|-------|------|------|
| **Eager Test** | 一个测试测了多个行为 | 一个测试里既有登录又有下单 |
| **Implementation Obsession** | 测试断言实现细节 | `verify(mock).called_once()` |
| **Lack of Isolation** | 测试之间有隐式依赖 | 先跑test_A才能跑test_B |
| **Brittle Test** | 实现小改测试就碎 | 依赖DOM结构或CSS选择器 |
| **Assertion Roulette** | 多个断言失败后不知道哪个错 | 一个测试里有10个assert |

## 低风险快速通道

对于简单的、低风险的变更，可以跳过完整的三Agent流程：

**适用场景**：
- 拼写错误修复
- 配置值调整
- 日志添加
- 注释更新
- 纯文档变更

**快速流程**：
1. 确认变更范围
2. 实现变更
3. 运行现有测试确保不破坏任何东西
4. 完成

**必须走完整流程的场景**：
- 任何新功能实现
- 任何业务逻辑变更
- 任何修复bug的代码变更
- 任何涉及API签名变更

## 与 property-based-testing 的配合

### 何时只用 adversarial-tdd

- 写新功能（行为测试）
- 修复 bug（回归测试）
- 重构非数据层代码

### 何时只用 property-based-testing

- 重构数据层（变换规则不破）
- 发现 flaky 测试（属性替代精确断言）
- 测试序列化/反序列化

### 何时两个都用

- **PydanticAI 项目**：先用 PBT 测 AI 输出的结构属性，再用 adversarial-tdd 测业务流程
- **新功能涉及数据模型**：先用 adversarial-tdd 测行为，再用 PBT 测数据模型的 roundtrip/不变量

### 执行顺序

```
写新功能：
  adversarial-tdd（行为测试）→ PBT（数据模型属性测试）

重构数据层：
  PBT（变换规则不破）→ adversarial-tdd（行为不破）

PydanticAI：
  PBT（结构属性）→ adversarial-tdd（业务流程）
```

两个技能的测试设计师可以合并（都是"攻击方"视角），审计员也可以复用。
