# 属性驱动测试（PBT）

## 概述

传统测试的思维是"我知道正确答案是什么，所以我来验证它"——你只能测你能想到的。

PBT 的思维是"我不知道所有正确答案，但我知道有些规则永远不能破"——框架自动生成成百上千个随机输入来替你找反例。

**核心转变**：从"我想几个例子测一下"变成"我描述代码必须遵守的规律，让机器来找违规的输入"。

## 思维方式：从例子到规律

考虑一个排序函数。传统思维是：

```
输入 [3,1,2] → 输出 [1,2,3] ✓
输入 [] → 输出 [] ✓
输入 [5] → 输出 [5] ✓
```

属性思维是：

```
规则1：排序后元素一个不少（长度不变、集合相等）
规则2：排序后升序（每个元素 ≤ 下一个）
规则3：再排一次不变（幂等）
```

三条规则，比一百个例子覆盖得更彻底。**例子验证过去，规则守护未来。**

## 何时触发

PBT 适合任何有"规律"的代码。以下信号出现时，主动建议 PBT：

| 信号 | 说明 |
|------|------|
| 有 encode + decode | 序列化对天然适合 roundtrip |
| 有 validate / is_valid | 验证器适合"合法输入不崩溃 + 非法输入正确拒绝" |
| 有 sort / filter / map / normalize | 数据变换有明确的不变量 |
| 有纯函数 / 数学运算 | 交换律、结合律、单位元 |
| 有 add/remove 操作 | 逆操作 |
| 有新旧两版实现 | Oracle 对比 |
| 涉及 AI/LLM 输出 | 无法断言精确值，但可以断言结构属性 |
| 用户说"边界条件怎么测" | PBT 自动发现边界 |
| 用户提到 PBT / Hypothesis / 属性测试 | 直接触发 |

## 如何发现属性

拿到一段代码，问自己三个问题：

**问题1：这个操作的反操作是什么？**
→ 做一遍再做回来，应该回到原点（Roundtrip、Inverse）

**问题2：什么属性在操作前后永远不变？**
→ 长度、集合、类型、值的范围（Invariant）

**问题3：如果做两遍会怎样？以不同顺序做会怎样？**
→ 结果不变、顺序无关（Idempotence、Commutativity）

不需要记住 10 种模式的名字。需要的时候查 `references/01-属性模式目录.md`，里面有每种模式的判断方法和示例。

## 核心工作流

### 阶段1：发现属性

扫描目标代码，为每个函数/类写出至少两条属性规则。用自然语言写，不需要代码。

```
目标：User.toJson() / User.fromJson()
属性1：任何 User 对象，toJson 再 fromJson 应该回到同一个 User
属性2：toJson 的输出必须是合法的 JSON 字符串
```

### 阶段2：写出测试

用对应语言的 PBT 工具把属性翻译成测试代码。具体语法查 `references/02-技术栈适配.md`。

### 阶段3：解读结果

PBT 失败时，框架会告诉你**最小反例**——不是第一个随机碰到的输入，而是自动缩小到的最简形式。比如一个让代码崩溃的字符串，框架会从 500 个字符一路缩到 `"*"` 或 `""`。

拿到最小反例后：
- 如果确实是 bug → 修复代码
- 如果反例不合法 → 说明你的策略生成了不该生成的输入，需要过滤
- 修复后把反例固定为回归用例

## 与 adversarial-tdd 配合

| 场景 | 先用 | 再用 |
|------|------|------|
| 写新功能 | adversarial-tdd（行为测试） | PBT（数据模型/变换的属性测试） |
| 重构数据层 | PBT（变换规则不破） | adversarial-tdd（行为不破） |
| PydanticAI 项目 | PBT（结构属性） | adversarial-tdd（业务流程） |
| 发现 flaky 测试 | PBT（属性替代精确断言） | — |

两个技能的测试设计师可以合并（都是"攻击方"视角），审计员也可以复用。

## 禁止模式

- 把 PBT 当成"自动生成测试用例的工具"——它是验证规律的工具，不是替代思考的工具
- 属性描述实现细节（"调用了一次 save"）——属性描述行为规律（"保存后再读取，数据一致"）
- 忽略最小反例，直接改测试让它通过
- 对 AI 输出用精确值断言
- 只发现属性不写测试，或只写测试不先想属性
