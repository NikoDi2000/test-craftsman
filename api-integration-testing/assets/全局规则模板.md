# API 集成测试全局规则

本文件定义了所有 Agent 在进行 API 集成测试时必须遵守的铁律。

## 核心原则

### 1. 测试隔离

- 每个测试必须独立运行，不依赖其他测试的状态
- 使用 `scope="function"` 的 fixture
- 测试结束后必须清除 `dependency_overrides`
- 数据库使用事务回滚，确保测试间隔离
- 使用 `autouse=True` 的清理 fixture 防止遗漏

### 2. Mock 边界

**必须 Mock**：
- 外部 HTTP 服务
- 第三方 API 调用
- 认证服务（使用 dependency_overrides）
- 邮件服务、通知服务

**禁止 Mock**：
- 数据库（使用真实 DB + 事务回滚）
- FastAPI 路由函数（使用 dependency_overrides 而非 patch）
- Pydantic 验证逻辑（测试真实行为）

### 3. 断言范围

**必须验证**：
- HTTP 状态码
- 响应体关键字段
- 数据库状态（关键场景）
- 错误消息格式

**禁止断言**：
- 内部实现细节
- 调用次数/顺序
- 私有方法
- 精确的时间戳（除非业务需要）

## 测试数据规则

### 命名规范

```python
# 好的命名
@pytest.fixture
def admin_user():
    """返回管理员用户"""
    return User(role="admin")

@pytest.fixture
def auth_headers(admin_user):
    """返回包含认证 token 的 headers"""
    return {"Authorization": f"Bearer {create_token(admin_user)}"}

# 不好的命名
@pytest.fixture
def u1():
    return User(id=1)

@pytest.fixture
def h1():
    return {"Authorization": "Bearer ..."}
```

### 测试数据原则

- 使用真实的数据类型（而非 "test1"、"test2"）
- 避免在测试中硬编码敏感信息
- 测试数据应该是确定性的（可重复运行）

## HTTP 语义规则

### 方法语义

| 方法 | 必须验证 |
|------|---------|
| GET | 不产生副作用（安全性） |
| POST | 创建返回 201 + Location |
| PUT | 全量替换（幂等性） |
| DELETE | 删除已删除返回 204/404（幂等性） |
| PATCH | 部分更新 |

### 状态码精确性

- 201 不是 200（创建 vs 成功）
- 401 不是 403（未认证 vs 权限不足）
- 401 必须返回 `WWW-Authenticate` 头
- 400 vs 422（语法错误 vs 语义错误）
- 405 必须返回 `Allow` 头

## 安全测试规则

### 必须覆盖

| 类别 | 测试项 | 优先级 |
|------|--------|--------|
| BOLA | 跨用户数据访问 | P0 |
| 注入 | SQL/NoSQL/命令注入 | P0 |
| 认证绕过 | Token 篡改/伪造 | P0 |
| 数据暴露 | 密码不在响应中 | P0 |
| 速率限制 | 超限返回 429 | P1 |
| 配置安全 | 生产禁用 /docs | P1 |

### 敏感信息不泄露

- 500 错误不应包含 SQL 语句、表名、堆栈跟踪
- 响应不应包含 password、password_hash
- CORS 不应配置为 `*`

## 数据库测试规则

### 事务回滚

```python
@pytest.fixture(scope="function")
def db_session():
    session = TestingSessionLocal()
    transaction = session.begin_nested()
    yield session
    session.rollback()
    session.close()
```

### 数据验证

```python
def test_user_created_in_db(db_session):
    response = client.post("/users/", json={"name": "Bob", "email": "bob@example.com"})
    assert response.status_code == 201

    user = db_session.query(User).filter_by(email="bob@example.com").first()
    assert user is not None
    assert user.name == "Bob"
```

## 异步测试规则

### AsyncClient 使用

```python
@pytest.mark.asyncio
async def test_async_endpoint():
    async with AsyncClient(
        transport=ASGITransport(app=app),
        base_url="http://test"
    ) as ac:
        response = await ac.get("/async-data")
    assert response.status_code == 200
```

### 禁止

- 禁止在 `async with` 块外使用 AsyncClient
- 禁止用 TestClient 测试 async 路由的并发行为
- 禁止混用 TestClient 和 AsyncClient

## CI/CD 规则

### 必须通过的条件

- 所有测试在 CI 中必须通过
- 测试必须可重复运行
- 测试必须在本地和 CI 中产生一致的结果

### 测试速度

- 单元测试：< 1秒
- 集成测试：< 10秒
- 超过 10 秒的测试需要优化或考虑分层

## 禁止模式清单

| 模式 | 原因 | 替代方案 |
|------|------|----------|
| 使用 `patch` mock 路由函数 | FastAPI 在导入时注册路由 | 使用 `dependency_overrides` |
| 测试间共享数据库状态 | 导致 flaky tests | 事务回滚 |
| 在测试中 commit | 破坏测试隔离 | 使用 savepoint |
| 测试真实外部服务 | 不可靠、慢 | 使用 pytest-httpx mock |
| 断言实现细节 | 测试脆弱 | 断言行为 |
| 使用 `scope="module"` 的数据库 fixture | 状态污染 | `scope="function"` |
| 使用 `datetime.utcnow()` | Python 3.12 已弃用 | `datetime.now(timezone.utc)` |
| 在 `async with` 外使用 AsyncClient | 连接已关闭 | 所有请求在 `async with` 块内 |
